  Architecture

  Config (config.py):
  - NestSimulationFromSonataScanConfig / NestSimulationFromSonataSingleConfig - takes a path to
  simulation_config.json and a configurable NEST point neuron model (default iaf_psc_alpha)

  Task (task.py):
  - NestSimulationFromSonataTask - reads the SONATA config, builds a NEST network from the circuit HDF5 files,
  applies stimuli/recordings/manipulations via handlers, simulates, and writes SONATA-compatible spike +
  voltage output

  Execution flow

  1. Parse simulation_config.json + circuit_config.json + node_sets.json
  2. Build NEST network: biophysical populations → point neurons, virtual → spike_generator, edges →
  static_synapse connections with delays/weights from HDF5
  3. Set V_m from conditions.v_init; log warnings for NEURON-specific conditions
  4. Apply each inputs entry via the appropriate stimulus handler
  5. Apply each reports entry via the appropriate recording handler; always add a spike recorder
  6. Log warnings for unsupported connection_overrides and modifications
  7. Simulate for run.tstop ms
  8. Write spikes.h5 and voltage.h5 in SONATA format

  Note: "core_nll_7" was not found in the codebase. The implementation works with any SONATA circuit - you can
  point it at any simulation_config.json generated by generate-simulations.







  Notebook created at examples/J_nest_simulation/nest_simulation_from_sonata.ipynb. It:

  1. Generates a campaign - Builds circuit_config.json and simulation_config.json pointing to core_nll_7 with
  absolute paths, 50ms duration, LGN + BKG spike inputs, a constant current stimulus (0.05 nA, 10-40ms window),
   and soma voltage recording
  2. Runs the NEST simulation - Creates a NestSimulationFromSonataSingleConfig, instantiates
  NestSimulationFromSonataTask, and executes
  3. Plots results - Spike rasters and voltage traces from the SONATA HDF5 output

  I also updated the task for core_nll_7 compatibility:
  - Embedded networks: _load_configs now handles configs where networks is in the simulation config directly
  (no separate circuit_config.json needed)
  - BMTK edge naming: _parse_edge_population_names now handles source_to_target format alongside
  source__target__type
  - BMTK node discovery: _discover_populations_from_hdf5 reads population info from HDF5 + CSV when no
  populations dict is present
  - Vectorized connections: Edge building uses numpy array indexing (src_global[source_ids]) instead of
  per-edge Python loops - critical for the 56M+ edges in core_nll_7
